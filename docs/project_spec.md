# สรุปโปรเจค: Dashboard คะแนน 5S Real Time (รังสิต)

เอกสารนี้รวบรวมข้อมูลภาพรวมของระบบจัดการและแสดงผลคะแนน 5S รวมถึงหลักการคำนวณและสูตรที่ใช้ในระบบ

## 1. ภาพรวมโปรเจค
โปรเจคนี้คือระบบ Dashboard แสดงผลคะแนน 5S แบบ Real-time ออกแบบมาเพื่อแสดงผลบนหน้าจอขนาด 1920x1080 ในแนวนอน โดยแบ่งการแสดงผลเป็น 3 คอลัมน์ และมีหน้าจัดการสำหรับระบุจำนวนพนักงานในแต่ละ Area เพื่อนำไปใช้เป็นฐานในการคำนวณคะแนน

## 2. ส่วนประกอบหลักของระบบ
- **Dashboard (`index.php`)**: หน้าจอแสดงผลหลัก แบ่งเป็น 3 ตาราง ตารางละ 10 รายการ (รวม 30 รายการ) เรียงลำดับจากคะแนนมากไปน้อย
- **หน้าจัดการพนักงาน (`manage_employees.php`)**: หน้าสำหรับกรอกจำนวนพนักงานแยกตาม Factory และ Office ในแต่ละ Area
- **ฐานข้อมูล JSON (`employee_counts.json`)**: ไฟล์สำหรับจัดเก็บข้อมูลจำนวนพนักงานที่ถูกกรอกมาจากหน้าจัดการ

## 3. หลักการคำนวณคะแนน (Calculation Methodology)

ระบบจะคำนวณคะแนนแยกกันระหว่างส่วนของ **Factory** และ **Office** ในแต่ละ Area

### 3. หลักการคำนวณคะแนนอย่างละเอียด (Technical Calculation Logic)

ระบบมีกระบวนการคำนวณข้อมูลโดยเริ่มตั้งแต่การดึงข้อมูลดิบจากฐานข้อมูล (SQL Server) จนถึงการประมวลผลเป็นเปอร์เซ็นต์ (PHP) ดังนี้:

### 3.1 การดึงข้อมูลจากตัวจัดเก็บ (Data Acquisition)
ระบบใช้คำสั่ง SQL (`UNION ALL`) เพื่อรวบรวมข้อมูลการประเมินจาก 2 แหล่งหลัก:
1. **งานโรงงาน**: ดึงจากตาราง `RSMSSQL.DAILY_FACTORY_5S`
2. **งานสำนักงาน**: ดึงจากตาราง `RSMSSQL.DAILY_OFFICE_5S`

**เงื่อนไขการดึงข้อมูล:**
- ดึงข้อมูลภายในช่วงวันที่ผู้ใช้งานกำหนด (StartDate - EndDate)
- กรองเฉพาะวันทำการ (จันทร์-ศุกร์) โดยใช้คำสั่ง `DATEPART(dw, updatedate)` เพื่อไม่นำวันเสาร์-อาทิตย์มาคำนวณ

### 3.2 ตัวแปรที่ใช้ในการคำนวณ (Variables Definitions)
ข้อมูลที่ได้จาก Query และไฟล์ JSON จะถูกนำมาแทนค่าในตัวแปรดังนี้:
- `factory_total_score` / `office_total_score`: ผลรวมคะแนนดิบที่คณะกรรมการให้จริง
- `f_emp` / `o_emp`: จำนวนพนักงานในหน่วยงาน (ดึงจากไฟล์ `employee_counts.json`)
- `f_qc` / `o_qc`: จำนวนข้อคำถามในแบบประเมิน 
- `working_days`: จำนวนวันทำงานจริงในช่วงวันที่เลือก (ไม่รวมวันหยุด)
- `5`: คะแนนเต็มสูงสุดต่อหนึ่งข้อคำถาม

### 3.3 ขั้นตอนการคำนวณ (Step-by-Step Formula)

**ขั้นตอนที่ 1: คำนวณคะแนนเต็มสูงสุดที่เป็นไปได้ (Possible Score)**
เป็นการหาว่าหากพนักงานทุกคนได้คะแนนเต็มทุกข้อในทุกวัน จะได้คะแนนเท่าไหร่
- `f_possible = f_emp * f_qc * 5 * working_days`
- `o_possible = o_emp * o_qc * 5 * working_days`

**ขั้นตอนที่ 2: คำนวณเปอร์เซ็นต์คะแนนแยกตามส่วน (Branch Score)**
- `Factory Score (%) = (factory_total_score / f_possible) * 100`
- `Office Score (%) = (office_total_score / o_possible) * 100`

**ขั้นตอนที่ 3: คำนวณคะแนนรวมพื้นที่ (Total Area Score)**
ระบบจะนำคะแนนจากขั้นตอนที่ 2 มาหาค่าเฉลี่ย โดยพิจารณาจากพนักงานที่มีตัวตนจริง:แ
- **กรณีมีทั้ง Factory และ Office**: `Total Score = (Factory Score + Office Score) / 2`
- **กรณีมีส่วนเดียว**: เช่น มีแค่ Office จะใช้ `Total Score = Office Score` ทันที (ไม่นำไปหาร 2 เพื่อไม่ให้คะแนนถูกดึงลดลงจากส่วนที่ไม่มีพนักงานจริง)
- **กรณีไม่มีพนักงานระบุ**: `Total Score = 0%`

## 4. รายละเอียดการดึงข้อมูลและเงื่อนไขการแสดงผล (Data & Display Logic)

เพื่อให้ข้อมูลมีความถูกต้องและสะท้อนผลงานจริง ระบบมีเงื่อนไขการดึงข้อมูลดังนี้:

### 4.1 ช่วงเวลาของข้อมูล (Data Timeframe)
- **วันปกติ (วันที่ 2 เป็นต้นไป):** ระบบจะแสดงผลคะแนนสะสมตั้งแต่วันที่ 1 ของเดือนปัจจุบัน จนถึง **"เมื่อวานนี้"** (`EndDate = Today - 1`)
- **วันเริ่มเดือนใหม่ (วันที่ 1):** เพื่อให้สรุปผลเดือนที่ผ่านมายังคงแสดงอยู่ ระบบจะดึงข้อมูลของ **"เดือนที่แล้วทั้งเดือน"** มาแสดงผลโดยอัตโนมัติ
- **การกรองวันทำงาน:** ระบบจะคำนวณคะแนนเฉพาะ **วันจันทร์ - ศุกร์** เท่านั้น (ไม่นำข้อมูลวันหยุดเสาร์-อาทิตย์มาคำนวณเพื่อไม่ให้ตัวหารผิดเพี้ยน)

### 4.2 ระบบการรีเฟรชข้อมูล (Auto-Refresh Mechanisms)
Dashboard ออกแบบมาสำหรับการเปิดทิ้งไว้บนจอ Monitor ตลอด 24 ชั่วโมง โดยมีระบบรีเฟรช 2 รูปแบบ:
1. **Live Update (ทุก 5 วินาที):** ระบบจะคอยตรวจสอบไฟล์ `employee_counts.json` และ `dashboard_config.json` หากมีการแก้ไขจำนวนพนักงานหรือการตั้งค่า UI หน้าจอจะรีเฟรชตัวเองทันทีภายใน 5 วินาที
2. **Scheduled Refresh (07:30 น.):** ทุกๆ เช้าเวลา 07:30 น. ระบบจะทำการรีเฟรชหน้าจอ 1 ครั้งโดยอัตโนมัติ เพื่อให้มั่นใจว่าข้อมูลวันใหม่และเดือนใหม่ได้รับการอัปเดตอย่างสมบูรณ์

## 5. คุณสมบัติเด่นของระบบ (Key Features)
- **การเรียงลำดับอัตโนมัติ (Auto-Sorting)**: ระบบจะเรียงลำดับ Area ที่มีคะแนนสูงสุดไว้ด้านบนเสมอ
- **การเน้นสีตามเงื่อนไข (Conditional Formatting)**: ระบบรองรับการตั้งค่าสีตามคะแนนได้ 2 รูปแบบหลัก:
    1. **Summary Standard (Fixed)**: ใช้เกณฑ์สีคงที่ (80, 70, 60, 50) ตามมาตรฐานหน้าสรุป โดยสีจะแสดงที่พื้นหลังของช่องคะแนนเท่านั้น
    2. **Custom (Row Conditions)**: กำหนดเงื่อนไขเองได้อย่างอิสระ (เช่น < 50%) พร้อมเลือกสีตัวอักษรและพื้นหลังได้ทั้งแบบทั้งแถว หรือแยกรายคอลัมน์ (No., Area, Score)
- **ระบบปรับแต่งผ่าน GUI (Theme Builder)**: ผู้ดูแลสามารถปรับแต่งสีหลัก (Title, Month, Header), ขนาดตัวอักษรแยกตามคอลัมน์ และระยะขอบต่างๆ ได้ผ่านหน้าจอ Settings โดยไม่ต้องแก้ไข Code

## 7. ประวัติการพัฒนา (Implementation History)
- **Phase 1**: วางโครงสร้าง Dashboard เบื้องต้น
- **Phase 2**: สร้างระบบจัดการพนักงานและเชื่อมต่อข้อมูลด้วย JSON
- **Phase 3**: ปรับปรุง Layout เป็นแบบ 3 คอลัมน์ในแนวนอน และยกเลิกการสลับหน้า (Pagination)
- **Phase 4**: ปรับปรุง UI, ขนาดตัวอักษร, และการแสดงผลให้เต็มหน้าจอ 1080p
- **Phase 5**: เพิ่มระบบ Live Update และการเชื่อมโยงระหว่างหน้าจอ Dashboard กับหน้าจัดการ
- **Phase 6**: เพิ่มระบบจัดการผ่าน Configuration (`dashboard_config.json`) และกล่องสรุปคะแนนเฉลี่ยรวม (Summary Score) 
- **Phase 7**: พัฒนาระบบ Theme Builder และ Scoring System Mode
    - **Scoring Modes**: เพิ่มการสลับโหมดระหว่าง "เกณฑ์คงที่ตาม Summary" (เน้นพื้นหลังช่องคะแนน) และ "เกณฑ์อิสระ" (เน้นทั้งแถว)
    - **Advanced Row/Column Styling**: เพิ่มความสามารถในการตั้งค่าสีพื้นหลัง (Background Code) แยกตามเงื่อนไขคะแนน ทั้งแบบทั้งแถวและแยกรายคอลัมน์
    - **Dynamic Configuration UI**: ปรับปรุงหน้า Settings ให้รองรับการเลือกสี (Color Pickers) และการเพิ่ม/ลดเงื่อนไขคะแนนผ่านหน้าจอโดยตรง
